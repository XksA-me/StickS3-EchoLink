# 功耗设计

M5StickS3 只有 250 mAh 电池，而实时语音设备又必须保持无线接收，因此本项目的目标是降低空闲功耗，不以牺牲实时消息为代价追求极限休眠。

## 已实施优化

### 显示

- 默认亮度 50/255。
- 15 秒无操作降到 8/255。
- 30 秒无操作调用显示 Sleep。
- 黑屏后 LifeTime 不再向 ST7789 提交画面。
- 普通模式按下 A/B 或收到新语音时唤醒；电源键单击进入监听后不会因收件自动亮屏。
- LifeTime 使用 PSRAM Canvas 整帧提交，避免清屏闪烁和多次 SPI 更新。
- EchoLink 动画降到 2 FPS；LifeTime 沙漏在运行或晃动时最多每 450 ms 更新一次，静止暂停时不刷新；互动脉冲最多每秒更新一次，木鱼功德动画只在点击后短暂刷新。

### 音频

- ES8311 麦克风只在按住 A 录音时启用。
- 扬声器与 AW8737 功放只在提示音和播放期间启用。
- 录音、播放或提示音结束后立即关闭音频链路。

音频待机关闭通常比微调 UI 动画更有价值，因为 Codec 和功放属于持续耗电外设。

### IMU

- 只有 LifeTime 的 Hourglass / Together 在前台时读取 IMU。
- 采样周期默认 100 ms，Battery 页、EchoLink 和省电监听不读取 IMU。
- LifeTime 在后台不更新 UI、IMU 或倒计时。

### CPU 与任务

- 主循环录音时 2 ms、发送时 5 ms、亮屏空闲 20 ms、黑屏空闲 40 ms 调度。
- MQTT Task 有发布任务或待接收数据时立即处理，连接空闲时按网页配置的 0.1-60 秒间隔检查。
- MQTT 未启用或 Wi-Fi 未连接时使用更长 FreeRTOS Delay。
- 串口 Heartbeat 从每 5 秒降低到每 30 秒。

### Wi-Fi

- 默认保留 ESP32 modem sleep。
- 若 MQTT 已成功连接后发生异常断链，固件会在本次启动期间关闭 modem sleep，以可靠性优先。
- ESP-NOW Local 模式必须保持 2.4 GHz Radio，无法像纯离线工具一样关闭无线。

### 省电监听

单击电源键进入省电监听：

- 强制关闭背光和显示，关闭 LED 与音频链路。
- Hourglass 倒计时归零，暂停 LifeTime UI、IMU、互动发送和后台发件。
- 保留 ESP-NOW 或 Wi-Fi/MQTT 收件、新语音队列和三声提示。
- 收到语音不会自动点亮屏幕；再次单击电源键恢复前台。

双击电源键仍由设备执行原有关机行为。

## 页面刷新清单

页面提交不是主循环频率。主循环空闲时约每 20ms 调度一次，但只有状态变化或到达以下条件才会向屏幕提交新帧：

| 页面/状态 | 自动刷新 |
| --- | --- |
| EchoLink READY | 不自动刷新，仅按键、网络状态变化或提示消息时刷新 |
| EchoLink 录音/收发/播放 | 每 500ms，显示必要的进度和动态图形 |
| Wi-Fi 配置状态 | 扫描/连接状态变化时刷新，设备提示文字最长约 8 秒变化一次 |
| LifeTime BATTERY | 电池数据最多每 60 秒检查一次，用户操作立即刷新 |
| LifeTime HOURGLASS | 未运行且无晃动时不刷新；运行或晃动时最多每 450 ms 刷新沙层和回流形状 |
| LifeTime TOGETHER | 用户点击、摇动或收到 Ping 时刷新，短暂反馈最多每秒刷新一次 |
| 省电监听 | 不提交任何显示帧 |

## 为什么不使用 Deep Sleep

Deep Sleep 会停止 MQTT Socket、ESP-NOW 接收、按键扫描和软件定时器更新。设备只能定时醒来检查，但 QoS 0 MQTT 消息不会在 Broker 中等待，短消息会直接错过。

因此当前策略是：显示和外设休眠，无线与最小后台任务保持在线。

## 续航评估方法

软件只能减少负载，最终续航应实测。建议分别测量：

1. LOCAL + 黑屏待机。
2. WIFI + MQTT 稳定连接 + 黑屏待机。
3. LifeTime Hourglass 倒计时亮屏。
4. 连续录音和播放。

使用 USB 功耗计或串联电流表记录平均电流。理论续航粗略值：

```text
续航小时 = 250 mAh / 平均电流 mA
```

电池标称容量、转换损耗、信号强度、重连次数、音量和背光都会让实际值更低。

## 可选的更激进模式

当前 5 秒检查档仍保持 Wi-Fi/MQTT 在线。未来若需要一分钟或更长间隔，应增加“省电收件箱模式”：设备定时连接服务器拉取持久化消息，其他时间 Deep Sleep。这需要服务器保存消息、设备 ACK 和补拉 API，不适合当前实时 QoS 0 协议直接启用。
